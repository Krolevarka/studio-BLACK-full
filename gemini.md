# 🌟 Системные Правила Проекта (studio-BLACK) для ИИ-ассистента

Этот файл — **высший приоритет**. Он содержит детерминированные алгоритмы, якорные термины и правила, которые агенты обязаны применять безотказно. Отклонения недопустимы.

## 1. Leading Words & Concepts

- **Device-Split**: Строгое физическое разделение UI на мобильные и десктопные файлы (например, `mobile/` и `desktop/`), которые рендерятся через глобальный компонент `<DeviceSwitch>`. Никаких громоздких медиа-запросов для структуры.
- **Strict-Cleanup**: Гарантированное уничтожение ВСЕХ таймеров, слушателей и анимаций. Код не готов, пока всё не очищено в `onBeforeUnmount`.
- **No-Any**: Абсолютный запрет на использование типа `any` в TypeScript.

## 2. Execution Steps

При написании, изменении или рефакторинге кода агент обязан следовать алгоритму:

1. **Архитектурный анализ:** 
   - Проверь необходимость **Device-Split**.
   - Убедись, что компонент отвечает ТОЛЬКО за View. Логика, стейты и математика выносятся в `composables/` или `utils/`.
   - Если компонент больше 150-200 строк — декомпозируй его.

2. **Имплементация (Strict-Stack):**
   - Используй исключительно Nuxt 4, Vue 3.5+, Composition API (`<script setup lang="ts">`).
   - Используй **No-Any** TypeScript для всех данных, пропсов, эвентов и стейтов (`useState`, `ref`).
   - Не используй жесткие пиксели (`px`) для структуры. Используй `rem`, `em`, `%`, `vh`, `vw`, `dvh`.
   - Если `<div>` работает как кнопка, добавь `role="button"`, `tabindex="0"`, `aria-label` и обработчик `@keydown.enter`.

3. **Оптимизация рендеринга (60fps):**
   - Проверяй флаг `needsHeavyAnimations` (из `useDeviceSwitch()`) перед тяжелыми эффектами.
   - Используй `needsHoverEffects` для отключения ховеров на тач-устройствах. Кликабельная зона — минимум 44x44px.
   - Защита от реактивности: используй `shallowRef` или обычные JS-переменные для математических данных, меняющихся каждый кадр.
   - Динамически применяй `will-change: transform, opacity` только на время активной анимации.

4. **Очистка (Completion Check):**
   - Выполни **Strict-Cleanup**. Это жесткий **Completion Criterion**.
   - *Проверка:* На каждый `setInterval`, `setTimeout`, `requestAnimationFrame`, `watch`, `window.addEventListener`, внутренний `useEventBus().on` и GSAP-анимацию/таймлайн/ScrollTrigger ДОЛЖЕН быть соответствующий вызов очистки в `onBeforeUnmount`.

5. **Оформление:**
   - Избегай глобальных селекторов. Используй `<style scoped>`.
   - Используй семантический HTML5 (`<main>`, `<article>`, `<section>`, `nav`). Одна `H1` на страницу.

## 3. Failure Modes (Частые ошибки ИИ)

Используй этот список для самодиагностики перед завершением ответа:

- **Premature completion (Утечки GSAP):** ИИ часто забывает вызвать `gsap.killTweensOf(shape)` или `timeline.kill()` в `onBeforeUnmount`. **Решение:** Проверять все реактивные объекты, участвующие в GSAP.
- **Premature completion (Утечки Event Bus):** Подписка через `on('event')` без соответствующего `off()` при размонтировании.
- **Duplication:** Использование `window.dispatchEvent` вместо выделенного внутреннего стейта или `useEventBus`.

## 4. Алгоритм общения с пользователем

1. **Точечные изменения:** Меняй только то, что просят. Сохраняй старые комментарии.
2. **Без "воды":** Рабочий код, без длинных философских рассуждений.
3. **Проактивный аудит:** Если видишь баг или нарушение **Strict-Cleanup** / **Device-Split** — исправь и кратко сообщи.
